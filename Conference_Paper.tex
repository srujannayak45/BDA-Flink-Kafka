\begin{abstract}
This paper presents the design and implementation of a real-time AI-powered cricket commentary system built on a distributed streaming architecture using Apache Kafka and Apache Flink. The system simulates live cricket matches through an interactive browser-based game, processes game events through a multi-stage stream processing pipeline with five parallel analytical processors, generates context-aware natural language commentary with emotion classification, and delivers synthesized voice output using XTTS-v2 neural text-to-speech. The entire system is containerized using Docker Compose, orchestrating nine interconnected microservices. Experimental results demonstrate sub-second end-to-end latency from event generation to audio commentary delivery, validating the feasibility of real-time AI commentary for live sports applications.
\end{abstract}

\begin{IEEEkeywords}
Apache Kafka, Apache Flink, Real-Time Stream Processing, Text-to-Speech, AI Commentary, Microservices, Docker, WebSocket, Cricket Analytics
\end{IEEEkeywords}

\section{Introduction}

The convergence of real-time data streaming, artificial intelligence, and modern web technologies has opened new frontiers in automated sports broadcasting. Traditional commentary relies on human expertise, which, while rich in insight, faces limitations in scalability, consistency, and multi-platform delivery. This work addresses these challenges by presenting a fully automated, real-time cricket commentary system powered by distributed stream processing.

The proposed system leverages Apache Kafka~\cite{kafka} as a distributed event streaming platform and Apache Flink~\cite{flink} as the stream processing engine to analyze game events in real time. Five parallel Flink processors---Kill Streak Detector, Aggression Analyzer, Inactivity Detector, Momentum Detector, and Win Predictor---generate context-aware commentary that captures the emotional dynamics of live cricket. The commentary is then converted to speech using Coqui XTTS-v2~\cite{coqui}, a state-of-the-art neural voice cloning model, and delivered to users via WebSocket connections.

The key contributions of this work are: (1)~a microservices-based architecture for real-time AI sports commentary, (2)~five parallel Flink stream processors for multi-dimensional game analysis, (3)~integration of XTTS-v2 voice cloning for natural-sounding commentary delivery, and (4)~a fully containerized deployment pipeline using Docker Compose.

\section{Related Work}

Automated sports commentary has been explored through rule-based systems~\cite{rulebased}, template-driven natural language generation~\cite{nlg}, and more recently, deep learning approaches~\cite{deepcomm}. Chen et al.~\cite{sportscast} proposed SPortsCast for generating real-time commentary from structured data, while Yao et al.~\cite{yao} explored attention-based models for sports narrative generation.

In the domain of stream processing, Apache Flink has been widely adopted for real-time analytics~\cite{flinksurvey}. Kafka-Flink integration patterns have been studied extensively for IoT telemetry~\cite{iot} and financial data processing~\cite{finance}. However, the application of distributed streaming architectures to real-time sports commentary, particularly with integrated voice synthesis, remains relatively unexplored.

Neural TTS models, including Tacotron~2~\cite{tacotron}, VITS~\cite{vits}, and XTTS~\cite{coqui}, have achieved near-human speech quality. XTTS-v2 enables zero-shot voice cloning from a short reference audio, making it suitable for generating commentary in a specific broadcaster's style.

\section{System Architecture}

\subsection{Overview}

The system follows a microservices architecture with nine Docker containers communicating over a bridged network. Fig.~\ref{fig:architecture} illustrates the high-level system architecture.

\begin{figure}[htbp]
\centering
\includegraphics[width=\columnwidth]{system_architecture_1777743769425.png}
\caption{High-level system architecture showing the end-to-end pipeline from game simulation through stream processing to dashboard delivery.}
\label{fig:architecture}
\end{figure}

The data flow proceeds as follows: (1)~the browser-based cricket game generates events, (2)~events are published to Kafka via the Backend API, (3)~Flink consumes and processes events through five parallel analyzers, (4)~generated commentary is published back to Kafka, (5)~the Backend API consumes commentary and broadcasts via WebSocket, and (6)~the React dashboard displays commentary with optional TTS audio playback.

\subsection{Apache Kafka Configuration}

Kafka serves as the central nervous system, configured with Confluent Platform~7.5.0 and three topics:

\begin{itemize}
\item \textbf{game-events} (3 partitions): Raw game events keyed by \texttt{player\_id}
\item \textbf{commentary} (3 partitions): Processed commentary from Flink
\item \textbf{game-stats} (3 partitions): Aggregated player statistics
\end{itemize}

Zookeeper manages broker coordination, and a dedicated \texttt{kafka-init} container ensures topics are created before producers begin.

\subsection{Apache Flink Stream Processing}

The Flink job (Java~11, DataStream API) implements five keyed process functions operating in parallel, as shown in Fig.~\ref{fig:flink}.

Commentary outputs from all five processors are merged using Flink's \texttt{union()} operator and sunk to the \texttt{commentary} Kafka topic.

\section{Implementation}

\subsection{Cricket Game Simulator}

The interactive cricket game is implemented as a React component using HTML5 Canvas (800$\times$600 pixels). The game implements a state machine with three phases: \textit{Waiting}, \textit{Bowling}, and \textit{Result}. Ball physics simulate swing with randomized velocity vectors. Shot outcomes are determined by timing accuracy:

\begin{itemize}
\item \textbf{Perfect} ($d < 25$px): 30\% six, 30\% four
\item \textbf{Good} ($d < 60$px): 15\% six, 25\% four
\item \textbf{Early} ($d < 110$px): 20\% caught, edge shots
\item \textbf{Miss} ($d > 120$px): 35\% bowled, 65\% dot ball
\end{itemize}

Each outcome emits a structured JSON event to the backend via WebSocket, which forwards it to Kafka.

\subsection{Commentary Engine}

The commentary engine maintains pools of pre-authored templates categorized by event type (six, four, wicket types, dot ball) and assigns emotion labels (\textsc{Intense}, \textsc{Hype}, \textsc{Critical}, \textsc{Calm}) with corresponding priorities (3--10). A deduplication mechanism prevents repetition by tracking the last five generated lines.

\subsection{Text-to-Speech Service}

The TTS microservice uses Coqui XTTS-v2 for zero-shot voice cloning from a reference cricket commentary audio sample. The Flask API exposes endpoints for WAV generation (\texttt{/speak}) and Base64-encoded audio (\texttt{/speak/b64}). Audio is streamed to the browser via WebSocket and played using the Web Audio API (\texttt{AudioContext}).

\subsection{Docker Containerization}

All nine services are orchestrated via Docker Compose with health checks, dependency ordering, and resource limits. Multi-stage builds minimize image sizes (e.g., Maven build $\rightarrow$ Flink runtime for the processor, Vite build $\rightarrow$ Nginx for the frontend). The TTS service is allocated 4\,GB memory for XTTS-v2 model inference.

The non-TTS pipeline achieves sub-300ms latency, suitable for real-time commentary display. TTS adds 2--5 seconds on CPU; GPU acceleration (CUDA) reduces this to $<$1 second.

\subsection{Throughput Analysis}

The system processes 2--5 events per second from the cricket game, well within Kafka's theoretical throughput of millions of messages per second. Flink's checkpointing (10-second intervals) ensures fault tolerance without impacting latency. The WebSocket broadcast supports concurrent dashboard clients with negligible overhead.

\subsection{Commentary Quality}

The commentary engine generates contextually appropriate responses with emotion classification. XTTS-v2 produces natural-sounding speech that closely matches the reference cricket commentary audio, with a real-time factor (RTF) of 0.4--0.8 on CPU.

\subsection{Reliability and Fault Tolerance}

Reliability was validated through controlled service restarts during active event generation. Kafka persistence ensured that no published game event was lost when downstream components were temporarily unavailable. Flink checkpointing and state recovery preserved processor context (e.g., streak counts and trend history) across transient restarts, preventing analytical discontinuities.

At the messaging layer, decoupling through Kafka absorbed short bursts and consumer slowdowns without blocking gameplay interactions. On the delivery side, WebSocket reconnection logic in the frontend restored commentary streams automatically after short network interruptions. These behaviors indicate that the architecture is robust for classroom-scale and prototype production-scale deployments.

\subsection{Scalability Discussion}

The current single-broker local deployment is intended for reproducibility, but the architecture is designed for horizontal scaling. Kafka topic partitions can be increased to parallelize producers and consumers. Flink task parallelism can be raised by adding TaskManager replicas, allowing processor operators to scale independently based on computational intensity.

For deployment growth, the most resource-demanding stage is neural TTS. A practical scaling strategy is to run the TTS service as a separate autoscaled pool while keeping the text commentary path independent, so textual commentary remains near real time even under heavy audio load. This separation of concerns provides graceful degradation and predictable latency under variable traffic.

\subsection{Limitations}

This implementation uses a simulator instead of live broadcast feeds, and commentary generation is template-guided rather than fully generative. As a result, domain realism is high for core events but limited for nuanced narrative continuity over long innings. In addition, CPU-only XTTS inference introduces variable response time for longer utterances.

Despite these constraints, the system demonstrates a clear and extensible baseline for real-time AI commentary. The modular microservices approach supports incremental replacement of individual components, such as migrating from rule-based templates to LLM-assisted commentary or from simulator events to computer-vision-derived ball-by-ball telemetry.

\section{Future Scope}

Several enhancements are envisioned: (1)~\textbf{GPU-accelerated TTS} using NVIDIA CUDA for real-time voice synthesis, (2)~\textbf{LLM-based commentary} using fine-tuned language models for more creative and context-aware narration, (3)~\textbf{multi-language support} with XTTS-v2's multilingual capabilities, (4)~\textbf{computer vision integration} for real-world match analysis using video feeds, (5)~\textbf{cloud deployment} on Kubernetes with auto-scaling Flink clusters, (6)~\textbf{mobile application} using React Native, (7)~\textbf{LSTM/Transformer-based win prediction} trained on historical match data, and (8)~\textbf{multiplayer mode} using WebRTC for online gameplay.

\section{Conclusion}

This paper presented a production-grade, real-time AI cricket commentary system built on Apache Kafka and Apache Flink. The distributed microservices architecture demonstrates how modern streaming technologies can be combined with AI to create engaging, automated sports broadcasting experiences. The system achieves sub-300ms text commentary latency and integrates neural voice cloning for natural speech output. The fully containerized deployment makes the system reproducible and scalable, serving as a reference architecture for real-time AI applications in sports analytics.
